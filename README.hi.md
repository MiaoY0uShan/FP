# FP

**पैच अंतिम रेखा नहीं है। प्रमाण है।**

[![Validate](https://github.com/MiaoY0uShan/FP/actions/workflows/validate.yml/badge.svg)](https://github.com/MiaoY0uShan/FP/actions/workflows/validate.yml)
[![Release](https://img.shields.io/github/v/release/MiaoY0uShan/FP)](https://github.com/MiaoY0uShan/FP/releases)
[![License: MIT](https://img.shields.io/badge/license-MIT-22c55e.svg)](LICENSE)

अधिकांश कोडिंग एजेंट प्रॉम्प्ट से सीधे पैच की ओर दौड़ते हैं। FP आपके एजेंट को वास्तविक कार्य खोजने, हर प्रत्यायोजन को सीमित करने, और ऐसे प्रमाण के साथ समाप्त करने के लिए बाध्य करता है जिसे एक मूल एजेंट स्वतंत्र रूप से सत्यापित कर सके।

यह पिछले रन से भी सीख सकता है। बस एक भाग्यशाली घटना को स्थायी कानून में बदलकर नहीं।

FP लक्ष्य से सक्रियण का अनुमान लगाता है: यह इंजीनियरिंग कार्य के लिए स्वचालित रूप से लोड होता है और सामान्य बातचीत या अन्य गैर-इंजीनियरिंग लक्ष्यों के लिए निष्क्रिय रहता है। `FP:` और `$fp` वैकल्पिक स्पष्ट आह्वान बने रहते हैं।

कोई डेमॉन नहीं। कोई डेटाबेस नहीं। कोई आवश्यक MCP सर्वर नहीं। इसे इंस्टॉल करें, अपने एजेंट को पुनः लोड करें, और सामान्य रूप से काम करें।

---

## आप इस स्थिति को जानते हैं

चार एजेंट एक ही फ़ाइलों को छूते हैं। एक सेवा को पुनः आरंभ करता है। दूसरा हरी बिल्ड की रिपोर्ट करता है। कोई भी उस फ़ोन को पुनः परीक्षण नहीं करता जो अब भी कनेक्ट नहीं हो सकता।

FP कार्य को एक सीमा, एक स्वामी, और एक प्रेक्षणीय अंतिम रेखा देता है।

```text
FP के बिना

कॉन्फ़िग संपादित करें -> सेवा पुनः आरंभ करें -> हरी स्थिति -> "समाप्त"

FP के साथ

वास्तविक क्लाइंट को पुनरुत्पादित करें
-> वांछित / उत्पन्न / प्रभावी स्थिति की तुलना करें
-> पहली विफल सीमा खोजें
-> सबसे छोटा अधिकृत परिवर्तन करें
-> वास्तविक क्लाइंट + नकारात्मक नियंत्रण + जीवनचक्र पुनः चलाएँ
-> प्रमाण रिकॉर्ड करें
```

दूसरा मार्ग लगभग पाँच मिनट तक अनुमान लगाने से धीमा है। यह दो दिनों तक उस अनुमान को डीबग करने से काफ़ी तेज़ है।

## यह कैसे काम करता है

```text
अनुरोध
-> वास्तविक जोखिम के अनुसार रूट करें
-> क्षेत्र, प्राधिकार और स्वीकृति को स्थिर करें
-> सीमित कार्य निष्पादित या प्रत्यायोजित करें
-> प्रेक्षित जाँचें चलाएँ
-> साक्ष्य बहीखाता मान्य करें
-> वैकल्पिक रूप से पुन: प्रयोज्य सीखने के उम्मीदवार का मूल्यांकन करें
```

छोटा कार्य छोटा रहता है। घटनाएँ परिष्करण से पहले सेवा बहाल करती हैं। अज्ञात कारण पैच से पहले निदान को ट्रिगर करते हैं। बहु-उपकरण ऑडिट किसी भी लक्ष्य को बदलने से पहले आधारभूत प्रमाण एकत्र करते हैं। वर्तमान बाहरी तथ्य एक संस्करण, स्रोत, ताज़गी का आधार और विश्वास सीमा रखते हैं।

कोड जोड़ने से पहले, FP एक छोटी पुन: उपयोग सीढ़ी पर चलता है:

```text
1. क्या इसका अस्तित्व आवश्यक है?        नहीं -> छोड़ें (YAGNI)
2. पहले से कोडबेस में है?             हाँ -> पुन: उपयोग करें
3. मानक पुस्तकालय करता है?            हाँ -> उपयोग करें
4. नेटिव प्लेटफ़ॉर्म सुविधा?          हाँ -> उपयोग करें
5. स्थापित निर्भरता?                  हाँ -> उपयोग करें
6. एक स्पष्ट पंक्ति पर्याप्त है?       हाँ -> एक पंक्ति लिखें
7. तभी                                -> न्यूनतम नया कोड जोड़ें जो काम करे
```

सुरक्षा, रोलबैक, सुलभता, डेटा अखंडता और आवश्यक प्रमाण हटाने योग्य "जटिलता" नहीं हैं।

## वितरित, अराजक नहीं

```text
मूल / समाकलक
|-- सीमित जाँच A              केवल-पठन
|-- सीमित जाँच B              केवल-पठन
|-- उम्मीदवार शिक्षार्थी        केवल-पठन, केवल प्रस्ताव
|-- अंध मूल्यांकनकर्ता          छिपा हुआ होल्डआउट + ओरेकल
|-- विनिर्देश समीक्षक           स्वतंत्र कार्य + सत्र
+-- समाकलन समीक्षक              स्वतंत्र कार्य + सत्र
             -> सीमित प्रमाण + निर्णय

एक लेखक -> मूल महत्वपूर्ण जाँचें पुनः चलाता है -> प्रामाणिक बहीखाता
```

प्रत्येक तार्किक उप-कार्य को एक कार्य-स्थानीय लिफ़ाफ़ा प्राप्त होता है: कार्य/सत्र/मूल IDs, लक्ष्य, संदर्भ संदर्भ, भूमिका, उपकरण, मूल और प्रत्यक्ष-मूल प्राधिकार सीमाएँ, निर्भरताएँ, फ़ाइलें/संसाधन, पुनरावृत्ति/प्रयास/समय/गहराई सीमाएँ, अविकल्प्यता कुंजी, आउटपुट बजट, मूल-केवल कलाकृति पथ, और रुकने की शर्त।

मशीन अनुबंध विश्वास करने के बजाय व्युत्पन्न करता है:

- मूल और निर्भरता DAG वैधता और सफल निर्भरता समय;
- स्थिर इनपुट-क्रम परिणाम, प्रेक्षित समवर्तीता, प्रयास, समय-सीमा और पूर्वज रद्दीकरण;
- मूल/प्रत्यक्ष-मूल प्राधिकार, रिपॉज़िटरी क्षेत्र और URL-क्षेत्र प्रतिच्छेदन;
- अद्वितीय लेखक स्वामित्व प्लस धारक/पथ/समय-बद्ध पट्टा मुक्ति प्रमाण;
- वास्तविक शब्द/बाइट सारांश आकार और आरक्षित मूल-स्वामित्व कलाकृति मूल;
- पृथक विनिर्देश, गुणवत्ता, रद्दीकरण, अविकल्प्यता, पट्टा, संदर्भ-पृथक्करण और समाकलन आदेश जो रन, उत्पादक, द्वार और कवर किए गए कार्यों से बँधे हैं।

पत्तियाँ प्रत्यायोजित नहीं कर सकतीं, प्रमाण-पत्र उपयोग नहीं कर सकतीं, तैनात नहीं कर सकतीं, बाहरी संदेश नहीं भेज सकतीं, स्मृति को बढ़ावा नहीं दे सकतीं, या सक्रिय स्थिति को बदल नहीं सकतीं। एक बूलियन जो कहता है "पट्टे जारी किए गए" इस बात का प्रमाण नहीं है कि वे जारी किए गए थे।

समानांतरता स्वतंत्र कार्य के लिए है। एक ही साझा वृक्ष को संपादित करने वाले दो एजेंट एक वितरित प्रणाली नहीं हैं। वे आशावाद के साथ एक मर्ज विरोध हैं।

## दुर्घटना को याद किए बिना सीखें

FP बाहरी नीति विकसित करता है: कौशल, स्कीमा, जाँच सूचियाँ और सीमित स्वचालन। यह मॉडल भार प्रशिक्षित करने या सांख्यिकीय सामान्यीकरण की गारंटी देने का दावा नहीं करता।

```text
एक प्रमाणित रन
-> अवलोकन

एक गंभीर मामला
-> अधिकतम, संकीर्ण समाप्त होने वाली छाया जाँच सूची

2-4 स्वतंत्र सकारात्मक मामले
-> एक छोड़ें -> अंध मूल्यांकन -> हर मामले को घुमाएँ

सभी तह + नियंत्रण + अपरिवर्तनीय + भविष्य छाया + रोलबैक पास
-> मूल-अनुमोदित सक्रिय उम्मीदवार
```

व्याख्याएँ, शोर संस्करण और एक सत्र से पाँच उप-एजेंट उपयोगी मजबूती जाँचें हैं। वे फिर भी एक स्वतंत्र अनुभव के रूप में गिने जाते हैं।

एक सक्रिय उम्मीदवार को चाहिए:

- भिन्न कार्य, सत्र, इनपुट और कार्य-परिवार पहचानों के साथ पुनर्गणना योग्य स्रोत-बहीखाता स्नैपशॉट हैश;
- हर सकारात्मक मामला एक बार होल्ड आउट किया गया जब नमूना छोटा हो, होल्डआउट संदर्भ उम्मीदवार से छिपा हो;
- एक ही अंध मूल्यांकनकर्ता से आधारभूत, उम्मीदवार और स्वतंत्र-ओरेकल प्रमाण;
- समान-इकाई माप जो कोई प्रतिगमन और कम से कम एक सुधार नहीं दिखाते;
- पास का एक मामला जहाँ नियम को ट्रिगर नहीं होना चाहिए;
- शून्य-सहिष्णुता प्राधिकार, क्षेत्र, सुरक्षा, रद्दीकरण और अविकल्प्यता अपरिवर्तनीय;
- एक जटिलता बजट और एक अनुप्रयुक्त लक्ष्य जिसके बाइट जमे हुए उम्मीदवार हैश से मेल खाते हैं;
- तीन वास्तविक रूप से बाद की छाया अवलोकन, प्रत्येक नई और एक विश्वसनीय बहीखाता घड़ी द्वारा सीमित;
- स्पष्ट पदोन्नति प्राधिकार, वर्तमान उद्गम और परीक्षित रोलबैक।

प्रशिक्षण विफलताएँ अल्प-अनुकूलन को उजागर करती हैं। होल्डआउट प्रतिगमन अति-अनुकूलन को उजागर करते हैं। नकारात्मक-नियंत्रण विफलताएँ अति-व्यापक ट्रिगर को उजागर करती हैं। किसी को भी औसत नहीं किया जा सकता।

देखें [सामान्यीकरण द्वार](fp/generalization-gate/SKILL.md) और इसका [मशीन अनुबंध](fp/contracts/evidence-ledger.v1.schema.json)।

## प्रमाण जो "नहीं" कह सके

`fp/contracts/evidence-ledger.v1.schema.json` प्लस शून्य-निर्भरता अर्थगत सत्यापक सत्य का स्रोत बनाते हैं।

वे दावों को प्रेक्षित आदेशों से बाँधते हैं, पृथक रिपॉज़िटरी/नेटवर्क/लेखन क्षेत्र लागू करते हैं, अंतिम रन की उसके संक्षिप्त विवरण से तुलना करते हैं, सक्रिय-प्रणाली और बाहरी-संदर्भ प्रमाण मान्य करते हैं, और असंबंधित जाँचों, निर्मित मीट्रिक, भविष्य-दिनांकित सीखने या अप्रचलित निरंतरता स्थिति पर बंद विफल होते हैं।

```text
क्षेत्र -> स्वीकृति पंक्तियाँ -> सीमित निष्पादन -> प्रेक्षित जाँचें -> सत्यापित दावे
```

एक हरी प्रक्रिया, सेवा पुनः आरंभ, उप-सारांश या कार्यान्वयन अंतर अपने आप में पूर्णता प्रमाण नहीं है। एक बार घोषित प्रमाण पास हो जाने पर, FP एक निर्णय जारी करता है और अपरिवर्तित स्थिति पर निर्णय-तटस्थ जाँचें जोड़ने के बजाय रुक जाता है।

## इंस्टॉल करें

एक संग्रह। एक इंस्टॉलर। एक केवल-पठन सत्यापन।

1. [Releases](https://github.com/MiaoY0uShan/FP/releases) से नवीनतम `fp-universal-v{version}.zip` डाउनलोड करें।
2. इसे प्रोजेक्ट रूट में निकालें।
3. Windows पर `INSTALL-FP.cmd` चलाएँ। macOS/Linux पर `sh ./INSTALL-FP.sh` चलाएँ।
4. Windows पर `INSTALL-FP.cmd -Verify` या macOS/Linux पर `sh ./INSTALL-FP.sh --verify` से सत्यापित करें, फिर AI उपकरण को पुनः लोड करें और सामान्य रूप से काम करें।

इंस्टॉलर लिखने से पहले स्वामित्व, टकराव, लिंक/पुनर्पार्स बिंदु, प्रबंधित ब्लॉक और बैकअप की जाँच करता है। सत्यापित अनइंस्टॉल केवल इंस्टॉलर-स्वामित्व वाली सामग्री को हटाता है।

इंजीनियरिंग लक्ष्य बिना कीवर्ड के FP सक्रिय करते हैं। ये स्पष्ट रूप वैकल्पिक बने रहते हैं:

```text

$fp इस रिपॉज़िटरी के रिलीज़ वर्कफ़्लो की बिना संपादन के समीक्षा करें।
```

FP एक प्रमाणित रन से सत्यापन दर, क्षेत्र विस्तार, पुनर्कार्य, संदर्भ-भार प्रॉक्सी और सत्यापित प्रगति प्रति टोकन की गणना कर सकता है। अनुपलब्ध मान `unknown` रहते हैं। एक निष्पक्ष तुलना कार्य, मॉडल, रिपॉज़िटरी संशोधन, प्राधिकार और स्वीकृति जाँचों को स्थिर करती है।

यहाँ कोई सजावटी "42% बेहतर" चार्ट नहीं है। सत्यापक पूछेगा कि आधाररेखा कहाँ गई।

## रूट

FP एक संपीड़ित 4-रूट मॉडल का उपयोग करता है:

| रूट | कब | क्या होता है |
| --- | --- | --- |
| **तत्काल / उच्च-दाँव** | घटनाएँ, कड़ी पूछताछ, प्रोटोकॉल परिवर्तन | इरादे की पुष्टि करें → प्राधिकार के भीतर कार्य करें। घटनाएँ मरम्मत से पहले बहाल करती हैं। |
| **केवल-पठन निदान** | अज्ञात विफलताएँ या सक्रिय स्कैन | डीबग-पहले: परिकल्पना → जाँच → अधिकृत सुधार। ऑडिट: प्रति-लक्ष्य आधाररेखा → P0/P1/P2 रिपोर्ट। |
| **निर्माण** | स्पष्ट या अस्पष्ट कार्यान्वयन | छोटा → लघु संक्षेप। मध्यम → निष्पादन संक्षेप + बहीखाता। अस्पष्ट → विचार कार्ड। बड़ा → न्यूनतम मॉड्यूल → अंतिम संक्षेप। |
| **समापन** | हर कार्य | मिलान प्रमाण के साथ पास → एक निर्णय → रुकें। अतिरिक्त निदान को एक निर्णय बदलना या एक स्वीकृति पंक्ति भरनी होगी; बदली हुई स्थिति फिर भी आवश्यक प्रतिगमन और नकारात्मक नियंत्रण ट्रिगर करती है। |

सूक्ष्म स्पष्ट संपादन → पाँच-पंक्ति संक्षेप और एक प्रासंगिक जाँच। सक्रिय आउटेज → `प्रेक्षण → नियंत्रण → बहाली → मरम्मत → सीखें`। बहु-उपकरण बेड़ा स्कैन → केवल-पठन प्रति-लक्ष्य प्रमाण, पार-लक्ष्य तुलना, फिर प्रति लक्ष्य अधिकृत मरम्मत।

सक्रिय प्रणालियाँ, बाहरी संदर्भ, प्रदाता संगतता/खर्च नियंत्रण, बहु-एजेंट और प्रत्यायोजित निष्पादन, निरंतरता, स्व-पुनरावृत्ति और पृष्ठभूमि सीखना इन रूटों पर परत के रूप में जुड़ते हैं।

प्रत्यायोजित निष्पादन अब स्पष्ट है: प्रत्येक कार्य मद को एक नया कार्यान्वयनकर्ता, नया कार्य समीक्षक, सशर्त नया सुधारक और नया पुनः-समीक्षक, फिर मूल द्वारा समाकलन जाँच पुनः चलाने से पहले एक नया अंतिम समाकलन समीक्षक मिलता है। स्वतंत्र समस्या डोमेन केवल तभी फैलते हैं जब साझा स्थिति, फ़ाइलें, उत्पन्न आउटपुट और निर्भरताएँ अलग सिद्ध हो जाएँ। FP मॉडल नाम से अनुमान लगाने के बजाय स्थापित होस्ट का प्रेक्षित नेटिव रनटाइम चुनता है — जैसे Codex `spawn_agent`, Claude Code `Agent`, Kimi Code `Agent`/`AgentSwarm`, या Qwen Code `agent`। होस्ट UI द्वारा दिखाए गए मानव नाम कॉस्मेटिक हैं; FP नियतात्मक कार्य IDs का उपयोग करता है।

तृतीय-पक्ष प्रदाताओं और स्थानीय गेटवे के लिए, प्रदाता-संगतता प्रोफ़ाइल प्रभावी होस्ट -> प्रॉक्सी -> वायर मॉडल -> प्रदाता श्रृंखला को हल करती है, प्रॉक्सी स्वास्थ्य जाँचती है, नेस्टेड पुनर्प्रयास सीमाओं को गुणा करती है, अनुरोध/टोकन/उप-एजेंट बजट को स्थिर करती है, दोहराए गए अर्थगत कार्यों को रोकती है, और प्रदाता-नेटिव उपयोग का मिलान करती है। यह असीमित खर्च को रोक सकती है और एक टूटी सीमा को इंगित कर सकती है; यह प्रोटोकॉल सुविधाएँ नहीं जोड़ सकती जो कोई प्रदाता लागू नहीं करता।

जब एक स्थापित MCP वास्तव में एक स्वीकृति पंक्ति के लिए आवश्यक हो, FP कार्य के मौजूदा प्राधिकार के भीतर स्वचालित रूप से इसका उपयोग करता है। यदि यह अनुपलब्ध है, FP पहले सटीक स्रोत, संस्करण, इंस्टॉल क्षेत्र, अनुमतियाँ/डेटा एक्सपोज़र, प्रमाणीकरण आवश्यकताएँ, सत्यापन और रोलबैक दिखाता है; उपयोगकर्ता द्वारा स्पष्ट रूप से अनुमोदित किए जाने तक कुछ भी डाउनलोड या इंस्टॉल नहीं किया जाता।

## विकास

प्रामाणिक स्रोत `fp/` में रहता है। उत्पन्न होस्ट पैक `install/` में 18 एजेंटों (Claude Code, Codex, Gemini CLI, Cursor, Windsurf, Cline, Roo Code, OpenCode, Kiro, Qoder, Aider, GitHub Copilot CLI, GitHub Copilot Editor और अधिक) के लिए रहते हैं। सभी उत्पन्न प्रतियाँ स्क्रिप्ट द्वारा ताज़ा की जाती हैं, कभी हाथ से संपादित नहीं।

```text
node scripts/lint-fp.js
node scripts/lint-release.js
node scripts/lint-contracts.js --ledger fp/examples/password-reset.evidence-ledger.json --brief fp/examples/password-reset.compiled-execution-brief.json
node --test
powershell -NoProfile -File scripts/sync-install-packs.ps1 -Check
```

रिलीज़ वर्कफ़्लो Windows PowerShell जीवनचक्र, POSIX इंस्टॉल/सत्यापन/अनइंस्टॉल, पैकेज चेकसम, संग्रह प्रवेश बिंदु और एक साफ उत्पन्न-पैक अंतर को भी गेट करता है।

## FAQ

### क्या हर कार्य एक समारोह बन जाता है?

नहीं। एक-पंक्ति के सुधार को संविधान की आवश्यकता नहीं होनी चाहिए।

### क्या उप-एजेंट पूरे कार्य को पूर्ण घोषित कर सकता है?

यह प्रमाण और निर्णय लौटा सकता है। मूल फिर भी महत्वपूर्ण जाँचें पुनः चलाता है और अंतिम दावे का स्वामी होता है।

### कुछ उपकरण एजेंटों के लिए विदेशी मानव नाम क्यों दिखाते हैं?

यह होस्ट UI लेबलिंग है, FP की आवश्यकता नहीं। FP `T03-implementer` और `T03-reviewer` जैसे अर्थगत IDs रिकॉर्ड करता है; रनटाइम पहचान, स्वामित्व, ताज़गी या प्रमाण बदले बिना कोई भी उपनाम प्रस्तुत कर सकता है।

### क्या DeepSeek, Kimi या Qwen API स्वयं उप-एजेंट प्रदान करते हैं?

नहीं। वे मॉडल प्रदाता हैं। Kimi Code और Qwen Code अपने स्वयं के उप-एजेंट उपकरणों के साथ एजेंट होस्ट हैं; Claude Code के पीछे उपयोग किया गया मॉडल API अभी भी Claude Code के रनटाइम का उपयोग करता है। आधिकारिक-स्रोत रजिस्ट्री होस्ट को केवल-मॉडल API से अलग करती है और स्थापित क्षमता अनुपलब्ध होने पर बंद विफल होती है।

### क्या यह एक सफल रन से सीखता है?

यह एक अवलोकन रिकॉर्ड करता है। एक घटना एक स्कीमा नहीं है।

### क्या यह स्वायत्त स्व-संशोधित AI है?

नहीं। पृष्ठभूमि एजेंट जमे हुए उम्मीदवार प्रस्तावित कर सकते हैं। स्वतंत्र मूल्यांकनकर्ता उनका परीक्षण करते हैं। पदोन्नति के लिए घोषित प्राधिकार, मशीन प्रमाण, भविष्य छाया विंडो और रोलबैक आवश्यक है।

### क्या इसे Hermes, Context7 या किसी अन्य चल रही सेवा की आवश्यकता है?

नहीं। उनके उपयोगी प्रोटोकॉल विचारों को एक पोर्टेबल स्थानीय अनुबंध में रूपांतरित किया गया। उनके डेमॉन, डेटाबेस, MCP सेवाएँ, निजी बैकएंड और क्रॉलर निर्भरताएँ नहीं हैं।

### क्या FP स्वचालित रूप से अनुपलब्ध MCP इंस्टॉल करेगा?

केवल स्पष्ट अनुमोदन के बाद। FP पहले से उपलब्ध कार्य-आवश्यक MCP को स्वचालित रूप से कॉल करता है, लेकिन अनुपलब्ध MCP को पहले एक सीमित अधिग्रहण संक्षेप मिलता है। इंस्टॉल की अनुमोदन चुपचाप लॉगिन, रहस्य, कॉन्फ़िगरेशन परिवर्तन, पुनः आरंभ, निवासी सेवाएँ या व्यापक कार्य क्रियाओं को अधिकृत नहीं करता।

### क्या यह कम टोकन या तेज़ डिलीवरी का वादा करता है?

केवल एक नियंत्रित दोहराई गई तुलना द्वारा सिद्ध होने के बाद।
