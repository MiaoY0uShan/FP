# FP

**Патч — не финишная черта. Доказательство — да.**

[![Validate](https://github.com/MiaoY0uShan/FP/actions/workflows/validate.yml/badge.svg)](https://github.com/MiaoY0uShan/FP/actions/workflows/validate.yml)
[![Release](https://img.shields.io/github/v/release/MiaoY0uShan/FP)](https://github.com/MiaoY0uShan/FP/releases)
[![License: MIT](https://img.shields.io/badge/license-MIT-22c55e.svg)](LICENSE)

Большинство программирующих агентов спешат от запроса к патчу. FP заставляет вашего агента найти настоящую задачу, ограничить каждое делегирование и завершить работу доказательством, которое родительский агент может проверить независимо.

Он также может учиться на предыдущих запусках. Но не превращая одну удачную случайность в вечный закон.

FP определяет активацию по цели: он загружается автоматически для инженерной работы и остаётся неактивным для обычного общения или других неинженерных целей. `FP:` и `$fp` остаются опциональными явными вызовами.

Без демона. Без базы данных. Без обязательного MCP-сервера. Установите, перезагрузите агента и работайте как обычно.

---

## Вам знакома эта ситуация

Четыре агента трогают одни и те же файлы. Один перезапускает сервис. Другой сообщает об успешной сборке. Никто не перепроверяет телефон, который всё ещё не может подключиться.

FP задаёт работе границу, владельца и наблюдаемую финишную черту.

```text
Без FP

изменить конфигурацию -> перезапустить сервис -> зелёный статус -> "готово"

С FP

воспроизвести поведение реального клиента
-> сравнить желаемое / сгенерированное / фактическое состояние
-> найти первый отказавший рубеж
-> внести минимальное авторизованное изменение
-> перезапустить реального клиента + отрицательный контроль + жизненный цикл
-> зафиксировать доказательство
```

Второй путь медленнее угадывания примерно на пять минут. Но значительно быстрее, чем отлаживать это угадывание два дня.

## Как это работает

```text
запрос
-> направить по фактическому риску
-> зафиксировать объём, полномочия и критерии приёмки
-> выполнить или делегировать ограниченную работу
-> провести наблюдаемые проверки
-> проверить Журнал Доказательств
-> опционально оценить кандидата на повторно используемое обучение
```

Малая работа остаётся малой. Инциденты восстанавливают сервис до полировки. Неизвестные причины запускают диагностику до патчей. Многоустройственные аудиты собирают базовые доказательства до любых изменений. Текущие внешние факты несут версию, источник, основание свежести и границу доверия.

Перед добавлением кода FP также проходит короткую лестницу повторного использования:

```text
1. Должно ли это существовать?              нет -> пропустить (YAGNI)
2. Уже есть в кодовой базе?                да -> использовать повторно
3. Стандартная библиотека это делает?       да -> использовать
4. Нативная функция платформы?              да -> использовать
5. Установленная зависимость?               да -> использовать
6. Достаточно одной строки?                 да -> написать одну строку
7. Только тогда                             -> добавить минимальный новый работающий код
```

Безопасность, откат, доступность, целостность данных и обязательные доказательства — не «сложность», которую можно удалить.

## Распределённо, а не хаотично

```text
родитель / интегратор
|-- ограниченное исследование A        только чтение
|-- ограниченное исследование B        только чтение
|-- кандидат-обучаемый                 только чтение, только предложение
|-- слепой оценщик                     скрытая контрольная выборка + оракул
|-- рецензент спецификации             независимая задача + сессия
+-- рецензент интеграции               независимая задача + сессия
             -> ограниченные доказательства + вердикты

один писатель -> родитель перезапускает критические проверки -> канонический журнал
```

Каждая логическая дочерняя задача получает локальный конверт задачи: ID задачи/сессии/родителя, цель, ссылки на контекст, роль, инструменты, потолки полномочий корневого и прямого родителя, зависимости, файлы/ресурсы, лимиты итераций/попыток/времени/глубины, ключ идемпотентности, бюджет вывода, путь к артефакту только для родителя и условие остановки.

Машинный контракт выводит, а не доверяет:

- валидность DAG родителя и зависимостей и успешное время зависимости;
- стабильные результаты порядка ввода, наблюдаемый параллелизм, попытки, тайм-аут и отмена предка;
- полномочия корневого/прямого родителя, область репозитория и пересечения URL-областей;
- уникальное владение писателем плюс доказательство освобождения аренды с указанием держателя/пути/времени;
- фактический размер резюме в словах/байтах и зарезервированный корень артефакта родителя;
- отдельные команды спецификации, качества, отмены, идемпотентности, аренды, изоляции контекста и интеграции, привязанные к запуску, производителю, шлюзу и охваченным задачам.

Листья не могут делегировать, использовать учётные данные, развёртывать, отправлять внешние сообщения, продвигать память или изменять живое состояние. Булево значение «аренда освобождена» не является доказательством того, что она была освобождена.

Параллелизм — для независимой работы. Два агента, редактирующих одно и то же общее дерево — не распределённая система. Это конфликт слияния с оптимизмом.

## Учиться, не запоминая случайность

FP развивает внешнюю политику: навыки, схемы, контрольные списки и ограниченную автоматизацию. Он не претендует на обучение весов моделей или гарантию статистического обобщения.

```text
один доказательный запуск
-> наблюдение

один серьёзный случай
-> максимум — узкий временный теневой контрольный список

2-4 независимых положительных случая
-> исключить один -> слепая оценка -> ротировать каждый случай

все фолды + контроли + инварианты + будущая тень + откат пройдены
-> утверждённый родителем активный кандидат
```

Перефразирования, шумовые варианты и пять подагентов из одной сессии — полезные проверки устойчивости. Они всё равно считаются одним независимым опытом.

Активный кандидат требует:

- пересчитываемые хеши снимков исходного журнала с различными идентификаторами задачи, сессии, ввода и семейства задач;
- каждый положительный случай исключён один раз при малой выборке, с контекстом исключения, скрытым от кандидата;
- базовые, кандидатские и независимо-оракульные доказательства от одного слепого оценщика;
- измерения в одних единицах, не показывающие регрессии и по крайней мере одно улучшение;
- близкий случай, где правило не должно срабатывать;
- инварианты нулевой терпимости к нарушениям полномочий, объёма, безопасности, отмены и идемпотентности;
- бюджет сложности и применённую цель, чьи байты соответствуют замороженному хешу кандидата;
- три действительно последующих теневых наблюдения, каждое новое и ограниченное доверенными часами журнала;
- явные полномочия на продвижение, текущее происхождение и проверенный откат.

Ошибки обучения выявляют недообучение. Регрессии контрольной выборки выявляют переобучение. Ошибки отрицательного контроля выявляют слишком широкий триггер. Ничто из этого нельзя усреднить.

См. [Шлюз Обобщения](fp/generalization-gate/SKILL.md) и его [машинный контракт](fp/contracts/evidence-ledger.v1.schema.json).

## Доказательство, способное сказать «нет»

`fp/contracts/evidence-ledger.v1.schema.json` плюс семантический валидатор без зависимостей образуют источник истины.

Они привязывают утверждения к наблюдаемым командам, обеспечивают раздельные области репозитория/сети/записи, сравнивают финальный запуск с его брифом, проверяют доказательства живой системы и внешнего контекста и отказывают закрыто при нерелевантных проверках, сфабрикованных метриках, обучении с будущей датой или устаревшем состоянии продолжения.

```text
объём -> строки приёмки -> ограниченное выполнение -> наблюдаемые проверки -> верифицированные утверждения
```

Зелёный процесс, перезапуск сервиса, дочернее резюме или diff реализации сами по себе не являются доказательством завершения. Как только заявленное доказательство пройдено, FP выдаёт один вердикт и останавливается, вместо добавления нейтральных к решению проверок на неизменённом состоянии.

## Установка

Один архив. Один установщик. Одна проверка только для чтения.

1. Скачайте последний `fp-universal-v{version}.zip` со страницы [Releases](https://github.com/MiaoY0uShan/FP/releases).
2. Извлеките его в корень проекта.
3. В Windows запустите `INSTALL-FP.cmd`. В macOS/Linux запустите `sh ./INSTALL-FP.sh`.
4. Проверьте с помощью `INSTALL-FP.cmd -Verify` в Windows или `sh ./INSTALL-FP.sh --verify` в macOS/Linux, затем перезагрузите AI-инструмент и работайте как обычно.

Установщик проверяет владение, коллизии, ссылки/точки повторной обработки, управляемые блоки и резервные копии перед записью. Проверенная деинсталляция удаляет только содержимое, принадлежащее установщику.

[Точные команды и уровни совместимости](INSTALL.md) | [Миграция с ZeroToHero или Xskill](MIGRATION.md) | [Альтернатива копированием](fp-copy-paste.md)

Пакет Claude Code включает `.claude/skills/fp/` для обнаружения навыков и `.claude/CLAUDE.md` для системной автоинъекции — тот же механизм, что использует Superpowers. Другие хосты получают точки входа для конкретных инструментов через универсальный установщик.

Инженерные цели активируют FP без ключевого слова. Эти явные формы остаются опциональными:

```text
FP: Диагностировать и исправить регрессию сброса пароля.
$fp Проверить релизный рабочий процесс этого репозитория без редактирования.
```

## Числа, когда они настоящие

**Нет базовой линии — нет заявления об улучшении.**

FP может рассчитать коэффициент верификации, расползание объёма, переработку, прокси контекстной нагрузки и Токены на Проверенный Прогресс из доказательного запуска. Отсутствующие значения остаются `unknown`. Честное сравнение фиксирует задачу, модель, ревизию репозитория, полномочия и проверки приёмки.

Здесь нет декоративного графика «на 42% лучше». Валидатор спросит, куда делась базовая линия.

## Маршруты

FP использует сжатую 4-маршрутную модель:

| Маршрут | Когда | Что происходит |
| --- | --- | --- |
| **Срочный / Высокие ставки** | Инциденты, жёсткие допросы, смена протоколов | Подтвердить намерение → действовать в рамках полномочий. Инциденты восстанавливают до ремонта. |
| **Диагностика только чтение** | Неизвестные сбои или проактивное сканирование | Сначала отладка: гипотеза → зонд → авторизованное исправление. Аудит: базовая линия на цель → отчёт P0/P1/P2. |
| **Сборка** | Ясная или расплывчатая реализация | Малое → Краткий бриф. Среднее → Исполнительный бриф + Журнал. Расплывчатое → Карточки идей. Крупное → минимальные модули → финальный бриф. |
| **Закрытие** | Каждая задача | Пройти с совпадающими доказательствами → один вердикт → стоп. Дополнительная диагностика должна изменить решение или заполнить строку приёмки; изменённое состояние всё равно запускает обязательную регрессию и отрицательный контроль. |

Крошечная ясная правка → бриф в пять строк и одна релевантная проверка. Активная авария → `НАБЛЮДАТЬ → ЛОКАЛИЗОВАТЬ → ВОССТАНОВИТЬ → ИСПРАВИТЬ → НАУЧИТЬСЯ`. Сканирование парка устройств → доказательства только для чтения на каждую цель, кросс-целевое сравнение, затем авторизованный ремонт на цель.

Живые системы, внешний контекст, совместимость провайдеров/контроль расходов, мультиагентное и делегированное исполнение, продолжение, самоитерация и фоновое обучение накладываются как профили на эти маршруты.

Делегированное исполнение теперь явное: каждый рабочий элемент получает свежего реализатора, свежего рецензента задачи, условного свежего исправителя и свежего перерецензента, затем свежего финального рецензента интеграции перед тем, как родитель перезапустит интеграционные проверки. Независимые проблемные домены разветвляются только после доказательства несвязанности общего состояния, файлов, сгенерированных выводов и зависимостей. FP выбирает наблюдаемое нативное рантайм-окружение установленного хоста — такое как Codex `spawn_agent`, Claude Code `Agent`, Kimi Code `Agent`/`AgentSwarm` или Qwen Code `agent` — вместо угадывания по имени модели. Человеческие имена, показываемые UI хоста, косметические; FP использует детерминированные ID задач.

Для сторонних провайдеров и локальных шлюзов профиль совместимости провайдеров определяет цепочку эффективный хост -> прокси -> проводная модель -> провайдер, проверяет здоровье прокси, умножает вложенные потолки повторных попыток, замораживает бюджеты запросов/токенов/подагентов, останавливает повторяющиеся семантические действия и сверяет нативное использование провайдера. Это может предотвратить неограниченные расходы и указать на сломанную границу; это не может добавить функции протокола, которые провайдер не реализует.

Когда установленный MCP действительно необходим для строки приёмки, FP использует его автоматически в рамках существующих полномочий задачи. Если он отсутствует, FP сначала показывает точный источник, версию, область установки, разрешения/доступ к данным, потребности аутентификации, проверку и откат; ничего не загружается и не устанавливается, пока пользователь явно не одобрит этот бриф.

## Разработка

Канонический исходный код находится в `fp/`. Сгенерированные хост-пакеты находятся в `install/` для 18 агентов (Claude Code, Codex, Gemini CLI, Cursor, Windsurf, Cline, Roo Code, OpenCode, Kiro, Qoder, Aider, GitHub Copilot CLI, GitHub Copilot Editor и других). Все сгенерированные копии обновляются скриптом и никогда не редактируются вручную.

```text
node scripts/lint-fp.js
node scripts/lint-release.js
node scripts/lint-contracts.js --ledger fp/examples/password-reset.evidence-ledger.json --brief fp/examples/password-reset.compiled-execution-brief.json
node --test
powershell -NoProfile -File scripts/sync-install-packs.ps1 -Check
```

Релизный рабочий процесс также проверяет жизненный цикл Windows PowerShell, установку/проверку/удаление POSIX, контрольные суммы пакетов, точки входа архива и чистый diff сгенерированного пакета.

## FAQ

### Каждая задача превращается в церемонию?

Нет. Исправление в одну строку не должно требовать конституции.

### Может ли подагент объявить всю задачу завершённой?

Он может вернуть доказательства и вердикт. Родитель всё равно перезапускает критические проверки и владеет финальным утверждением.

### Почему некоторые инструменты показывают иностранные человеческие имена для агентов?

Это маркировка UI хоста, а не требование FP. FP записывает семантические ID, такие как `T03-implementer` и `T03-reviewer`; рантайм может отображать любой псевдоним, не меняя идентичность, владение, свежесть или доказательства.

### Предоставляет ли DeepSeek, Kimi или Qwen API подагентов самостоятельно?

Нет. Это модельные провайдеры. Kimi Code и Qwen Code — это хосты агентов с собственными инструментами подагентов; модельный API, используемый за Claude Code, всё равно использует рантайм Claude Code. Официальный реестр источников различает хосты и API только для моделей и отказывает закрыто, когда установленная возможность отсутствует.

### Учится ли он на одном успешном запуске?

Он записывает одно наблюдение. Один случай — не схема.

### Это автономный самомодифицирующийся ИИ?

Нет. Фоновые агенты могут предлагать замороженных кандидатов. Независимые оценщики их тестируют. Продвижение требует заявленных полномочий, машинных доказательств, окна будущей тени и отката.

### Нужен ли ему Hermes, Context7 или другой запущенный сервис?

Нет. Их полезные протокольные идеи были адаптированы в портативный локальный контракт. Их демоны, базы данных, MCP-сервисы, частные бэкенды и краулеры не являются зависимостями.

### Установит ли FP отсутствующий MCP автоматически?

Только после явного одобрения. FP автоматически вызывает необходимый для задачи MCP, который уже доступен, но отсутствующий MCP сначала получает ограниченный бриф на получение. Одобрение установки не даёт молчаливого разрешения на логин, секреты, изменение конфигурации, перезапуски, резидентные сервисы или более широкие действия задачи.

### Обещает ли он меньше токенов или более быструю доставку?

Только после того, как контролируемое повторное сравнение это докажет.
