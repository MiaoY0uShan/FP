# FP

**التصحيح ليس خط النهاية. الدليل هو.**

[![Validate](https://github.com/MiaoY0uShan/FP/actions/workflows/validate.yml/badge.svg)](https://github.com/MiaoY0uShan/FP/actions/workflows/validate.yml)
[![Release](https://img.shields.io/github/v/release/MiaoY0uShan/FP)](https://github.com/MiaoY0uShan/FP/releases)
[![License: MIT](https://img.shields.io/badge/license-MIT-22c55e.svg)](LICENSE)

معظم عملاء البرمجة يندفعون من الطلب إلى التصحيح. FP يجعل عميلك يجد المهمة الحقيقية، ويحدد كل تفويض، وينتهي بدليل يمكن للعميل الأب التحقق منه بشكل مستقل.

يمكنه أيضًا التعلم من الجولات السابقة. لكن ليس بتحويل حكاية محظوظة إلى قانون دائم.

يستنتج FP التفعيل من الهدف: يُحمَّل تلقائيًا للعمل الهندسي ويبقى خاملًا للمحادثات العادية أو الأهداف غير الهندسية الأخرى. `FP:` و `$fp` يبقيان استدعاءات صريحة اختيارية.

لا خادم خلفي. لا قاعدة بيانات. لا خادم MCP مطلوب. ثبّته، وأعد تحميل عميلك، واعمل بشكل طبيعي.

## أنت تعرف هذا السيناريو

أربعة عملاء يلمسون نفس الملفات. أحدهم يعيد تشغيل الخدمة. آخر يبلغ عن بناء ناجح. لا أحد يعيد اختبار الهاتف الذي لا يزال غير قادر على الاتصال.

FP يمنح العمل حدًا ومالكًا وخط نهاية قابلًا للملاحظة.

```text
بدون FP

تعديل الإعدادات -> إعادة تشغيل الخدمة -> حالة خضراء -> "انتهى"

مع FP

إعادة إنتاج العميل الحقيقي
-> مقارنة الحالة المرغوبة / المولَّدة / الفعلية
-> إيجاد أول حد فاشل
-> إجراء أصغر تغيير مصرح به
-> إعادة تشغيل العميل الحقيقي + التحكم السلبي + دورة الحياة
-> تسجيل الدليل
```

المسار الثاني أبطأ من التخمين لحوالي خمس دقائق. إنه أسرع بكثير من تصحيح ذلك التخمين لمدة يومين.

## كيف يعمل

```text
الطلب
-> توجيه حسب المخاطر الفعلية
-> تجميد النطاق والصلاحية والقبول
-> تنفيذ أو تفويض عمل محدود
-> تشغيل فحوصات قابلة للملاحظة
-> التحقق من سجل الأدلة
-> اختياريًا تقييم مرشح تعلم قابل لإعادة الاستخدام
```

العمل الصغير يبقى صغيرًا. الحوادث تستعيد الخدمة قبل التحسين. الأسباب غير المعروفة تشغّل التشخيص قبل التصحيحات. عمليات التدقيق متعددة الأجهزة تجمع أدلة الأساس قبل تغيير أي هدف. الحقائق الخارجية الحالية تحمل إصدارًا ومصدرًا وأساس حداثة وحد ثقة.

قبل إضافة الكود، يسير FP على سلم إعادة استخدام قصير:

```text
1. هل يجب أن يوجد هذا؟                 لا -> تخطَّ (YAGNI)
2. موجود بالفعل في قاعدة الكود؟       نعم -> أعد استخدامه
3. المكتبة القياسية تفعل ذلك؟         نعم -> استخدمها
4. ميزة المنصة الأصلية؟               نعم -> استخدمها
5. اعتمادية مثبَّتة؟                   نعم -> استخدمها
6. سطر واحد واضح يكفي؟                نعم -> اكتب سطرًا واحدًا
7. عندها فقط                           -> أضف الحد الأدنى من الكود الجديد العامل
```

الأمان، والتراجع، وإمكانية الوصول، وسلامة البيانات، والأدلة المطلوبة ليست "تعقيدًا" يمكن حذفه.

## موزَّع، لا فوضوي

```text
الأب / المدمِج
|-- تحقيق محدود أ              للقراءة فقط
|-- تحقيق محدود ب              للقراءة فقط
|-- متعلم مرشح                 للقراءة فقط، اقتراح فقط
|-- مقيم أعمى                  مجموعة احتياط مخفية + وسيط
|-- مراجع المواصفات            مهمة وجلسة مستقلتان
+-- مراجع الدمج                مهمة وجلسة مستقلتان
             -> أدلة محدودة + أحكام

كاتب واحد -> الأب يعيد تشغيل الفحوصات الحرجة -> السجل القانوني
```

تتلقى كل مهمة فرعية منطقية ظرف مهمة محلي: معرّفات المهمة/الجلسة/الأب، الهدف، مراجع السياق، الدور، الأدوات، سقوف الصلاحية، الاعتماديات، الملفات/الموارد، حدود التكرار/المحاولة/الوقت/العمق، مفتاح العِوضية، ميزانية المخرجات، مسار الأثر الخاص بالأب فقط، وشرط التوقف.

العقد الآلي يستنتج بدلاً من أن يثق:

- صحة الرسم البياني للاعتماديات (DAG) للأب والاعتماديات وتوقيت الاعتمادية الناجح؛
- نتائج ترتيب الإدخال المستقرة، والتزامن الملاحظ، والمحاولات، والمهلة، وإلغاء الأصل؛
- صلاحية الأب/الأب المباشر، ونطاق المستودع، وتقاطعات نطاق URL؛
- ملكية كاتب فريدة بالإضافة إلى دليل تحرير العقد محدد الحامل/المسار/الوقت؛
- حجم الملخص الفعلي بالكلمات/البايتات وجذر أثر محجوز مملوك للأب؛
- أوامر منفصلة للمواصفات والجودة والإلغاء والعِوضية والعقد وعزل السياق والدمج مرتبطة بالجولة والمنتج والبوابة والمهام المغطاة.

الأوراق لا يمكنها التفويض، أو استخدام بيانات الاعتماد، أو النشر، أو إرسال رسائل خارجية، أو ترقية الذاكرة، أو تغيير الحالة الحية. قيمة منطقية تقول "تم تحرير العقود" ليست دليلاً على أنها حُرِّرت.

التوازي للعمل المستقل. وكيلان يحرران نفس الشجرة المشتركة ليسا نظامًا موزعًا. إنهما تعارض دمج مع تفاؤل.

## تعلم دون حفظ الحادث

يطور FP سياسة خارجية: المهارات، والمخططات، وقوائم التحقق، والأتمتة المحدودة. لا يدّعي تدريب أوزان النماذج أو ضمان التعميم الإحصائي.

```text
جولة موثقة واحدة
-> ملاحظة

حالة شديدة واحدة
-> قائمة تحقق ظل ضيقة منتهية الصلاحية، على الأكثر

2-4 حالات إيجابية مستقلة
-> اترك واحدة -> تقييم أعمى -> تدوير كل حالة

جميع الطيات + الضوابط + الثوابت + الظل المستقبلي + التراجع تجتاز
-> مرشح نشط معتمد من الأب
```

إعادة الصياغة، ومتغيرات التشويش، وخمسة وكلاء فرعيين من جلسة واحدة هي فحوصات متانة مفيدة. لا تزال تُحسب كتجربة مستقلة واحدة.

يحتاج المرشح النشط إلى:

- تجزئات لقطات سجل المصدر القابلة لإعادة الحساب بهويات مهمة وجلسة وإدخال وعائلة مهمة مميزة؛
- كل حالة إيجابية تُحتجز مرة واحدة عندما تكون العينة صغيرة، مع إخفاء سياق الاحتجاز عن المرشح؛
- أدلة الأساس والمرشح والوسيط المستقل من نفس المقيم الأعمى؛
- قياسات بنفس الوحدة لا تظهر أي تراجع وتحسنًا واحدًا على الأقل؛
- حالة قريبة لا يجب أن يشغّل فيها القاعدة؛
- ثوابت عدم التسامح مطلقًا مع الصلاحية والنطاق والأمان والإلغاء والعِوضية؛
- ميزانية تعقيد وهدف مطبَّق تطابق بايتاته تجزئة المرشح المجمد؛
- ثلاث ملاحظات ظل لاحقة حقًا، كل منها جديدة ومقيدة بساعة سجل موثوقة؛
- صلاحية ترقية صريحة، ومصدر حالي، وتراجع مختبَر.

فشل التدريب يكشف نقص الملاءمة. تراجعات الاحتجاز تكشف فرط الملاءمة. فشل الضوابط السلبية يكشف مشغِّلاً واسعًا جدًا. لا يمكن إلغاء أي منها بالمتوسط.

انظر [بوابة التعميم](fp/generalization-gate/SKILL.md) و[عقدها الآلي](fp/contracts/evidence-ledger.v1.schema.json).

## دليل يمكنه قول "لا"

`fp/contracts/evidence-ledger.v1.schema.json` بالإضافة إلى المدقق الدلالي بدون اعتماديات يشكلان مصدر الحقيقة.

يربطان الادعاءات بأوامر ملاحظة، ويفرضان نطاقات مستودع/شبكة/كتابة منفصلة، ويقارنان الجولة النهائية بموجزها، ويتحققان من أدلة النظام الحي والسياق الخارجي، ويفشلان بالإغلاق في الفحوصات غير ذات الصلة، أو المقاييس الملفقة، أو التعلم المؤرخ مستقبلاً، أو حالة الاستمرار القديمة.

```text
النطاق -> صفوف القبول -> تنفيذ محدود -> فحوصات ملاحظة -> ادعاءات محققة
```

العملية الخضراء، أو إعادة تشغيل الخدمة، أو ملخص الوكيل الفرعي، أو فارق التنفيذ ليس دليل إكمال بحد ذاته. بمجرد اجتياز الدليل المعلن، يصدر FP حكمًا واحدًا ويتوقف بدلاً من إضافة فحوصات محايدة القرار على حالة غير متغيرة.

## التثبيت

أرشيف واحد. مثبِّت واحد. تحقق واحد للقراءة فقط.

1. حمِّل أحدث `fp-universal-v{version}.zip` من [Releases](https://github.com/MiaoY0uShan/FP/releases).
2. استخرجه في جذر المشروع.
3. على Windows، شغِّل `INSTALL-FP.cmd`. على macOS/Linux، شغِّل `sh ./INSTALL-FP.sh`.
4. تحقق باستخدام `INSTALL-FP.cmd -Verify` على Windows أو `sh ./INSTALL-FP.sh --verify` على macOS/Linux، ثم أعد تحميل أداة الذكاء الاصطناعي واعمل بشكل طبيعي.

يتحقق المثبِّت من الملكية والتعارضات والروابط/نقاط إعادة التحليل والكتل المدارة والنسخ الاحتياطية قبل الكتابة. يزيل إلغاء التثبيت الموثق المحتوى المملوك للمثبِّت فقط.

[الأوامر الدقيقة ومستويات التوافق](INSTALL.md) | [الترحيل من ZeroToHero أو Xskill](MIGRATION.md) | [بديل النسخ واللصق](fp-copy-paste.md)

تتضمن حزمة Claude Code المجلد `.claude/skills/fp/` لاكتشاف المهارات و `.claude/CLAUDE.md` للحقن التلقائي على مستوى النظام — نفس الآلية التي تستخدمها Superpowers. يحصل المضيفون الآخرون على نقاط دخول خاصة بالأداة عبر المثبِّت الشامل.

الأهداف الهندسية تفعّل FP بدون كلمة مفتاحية. تبقى هذه الصيغ الصريحة اختيارية:

```text
FP: شخِّص وأصلح تراجع إعادة تعيين كلمة المرور.
$fp راجع سير عمل الإصدار لهذا المستودع دون تحرير.
```

## الأرقام، عندما تكون حقيقية

**لا خط أساس يعني لا ادعاء بتحسين.**

يمكن لـ FP حساب معدل التحقق، وتمدد النطاق، وإعادة العمل، ووكيل حمل السياق، والرموز إلى التقدم الموثق من جولة موثقة. تبقى القيم المفقودة `unknown`. المقارنة العادلة تثبِّت المهمة، والنموذج، ومراجعة المستودع، والصلاحية، وفحوصات القبول.

لا يوجد هنا رسم بياني تزييني يقول "أفضل بنسبة 42%". سيسأل المدقق أين ذهب خط الأساس.

## المسارات

يستخدم FP نموذجًا مضغوطًا من 4 مسارات:

| المسار | متى | ماذا يحدث |
| --- | --- | --- |
| **عاجل / عالي المخاطر** | الحوادث، والاستجوابات، وتغييرات البروتوكول | تأكيد النية → التصرف ضمن الصلاحية. الحوادث تستعيد قبل الإصلاح. |
| **تشخيص للقراءة فقط** | أعطال غير معروفة أو فحوصات استباقية | التشخيص أولاً: فرضية → مسبار → إصلاح مصرح به. التدقيق: خط أساس لكل هدف → تقرير P0/P1/P2. |
| **بناء** | تنفيذ واضح أو غامض | صغير → موجز مصغر. متوسط → موجز تنفيذ + سجل. غامض → بطاقات أفكار. كبير → وحدات دنيا → موجز نهائي. |
| **إغلاق** | كل مهمة | نجاح بأدلة متطابقة → حكم واحد → توقف. التشخيص الإضافي يجب أن يغير قرارًا أو يملأ صف قبول؛ الحالة المتغيرة لا تزال تشغّل التراجع والضوابط السلبية المطلوبة. |

تحرير صغير واضح → موجز من خمسة أسطر وفحص واحد ذي صلة. انقطاع نشط → `لاحظ → احتواء → استعادة → أصلح → تعلم`. فحص أسطول متعدد الأجهزة → أدلة لكل هدف للقراءة فقط، مقارنة عبر الأهداف، ثم إصلاح مصرح به لكل هدف.

الأنظمة الحية، والسياق الخارجي، وتوافق المزود/التحكم في الإنفاق، والتنفيذ متعدد الوكلاء والمفوض، والاستمرار، والتكرار الذاتي، والتعلم في الخلفية تطبَّق كطبقات على هذه المسارات.

التنفيذ المفوض أصبح الآن صريحًا: كل عنصر عمل يحصل على منفِّذ جديد، ومراجع مهام جديد، ومصحِّح جديد مشروط ومراجع إعادة جديد، ثم مراجع دمج نهائي جديد قبل أن يعيد الأب تشغيل فحوصات الدمج. مجالات المشكلات المستقلة تتفرع فقط بعد إثبات انفصال الحالة المشتركة والملفات والمخرجات المولَّدة والاعتماديات. يختار FP وقت التشغيل الأصلي الملاحظ للمضيف المثبَّت — مثل Codex `spawn_agent`، أو Claude Code `Agent`، أو Kimi Code `Agent`/`AgentSwarm`، أو Qwen Code `agent` — بدلاً من التخمين من اسم النموذج. الأسماء البشرية التي تظهرها واجهة المضيف تجميلية؛ يستخدم FP معرّفات مهام حتمية.

للمزودين الخارجيين والبوابات المحلية، يحلل ملف توافق المزود سلسلة المضيف الفعال -> الوكيل -> نموذج السلك -> المزود، ويتحقق من صحة الوكيل، ويضاعف سقوف إعادة المحاولة المتداخلة، ويجمِّد ميزانيات الطلب/الرموز/الوكلاء الفرعيين، ويوقف الإجراءات الدلالية المتكررة، ويسوي استخدام المزود الأصلي. يمكنه منع الإنفاق غير المحدود وتحديد حد معطل؛ لا يمكنه إضافة ميزات بروتوكول لا ينفذها المزود.

عندما يكون MCP مثبَّت مطلوبًا حقًا لصف قبول، يستخدمه FP تلقائيًا ضمن صلاحية المهمة الحالية. إذا كان مفقودًا، يعرض FP أولاً المصدر الدقيق، والإصدار، ونطاق التثبيت، والصلاحيات/تعرض البيانات، واحتياجات المصادقة، والتحقق، والتراجع؛ لا يُنزَّل أو يثبَّت أي شيء حتى يوافق المستخدم صراحةً على ذلك الموجز.

## التطوير

المصدر القانوني موجود في `fp/`. حزم المضيف المولَّدة موجودة في `install/` لـ 18 وكيلاً (Claude Code، Codex، Gemini CLI، Cursor، Windsurf، Cline، Roo Code، OpenCode، Kiro، Qoder، Aider، GitHub Copilot CLI، GitHub Copilot Editor والمزيد). جميع النسخ المولَّدة تُحدَّث بالبرنامج النصي، ولا تُحرر يدويًا أبدًا.

```text
node scripts/lint-fp.js
node scripts/lint-release.js
node scripts/lint-contracts.js --ledger fp/examples/password-reset.evidence-ledger.json --brief fp/examples/password-reset.compiled-execution-brief.json
node --test
powershell -NoProfile -File scripts/sync-install-packs.ps1 -Check
```

يتحكم سير عمل الإصدار أيضًا في دورة حياة Windows PowerShell، وتثبيت/تحقق/إلغاء تثبيت POSIX، والمجموع الاختباري للحزم، ونقاط دخول الأرشيف، وفارق حزم مولدة نظيف.

## الأسئلة الشائعة

### هل تصبح كل مهمة مراسم؟

لا. إصلاح من سطر واحد لا ينبغي أن يحتاج إلى دستور.

### هل يمكن للوكيل الفرعي أن يعلن إكمال المهمة بأكملها؟

يمكنه إرجاع الأدلة والحكم. الأب لا يزال يعيد تشغيل الفحوصات الحرجة ويملك الادعاء النهائي.

### لماذا تظهر بعض الأدوات أسماء بشرية أجنبية للوكلاء؟

هذا تصنيف من واجهة المضيف، وليس متطلبًا من FP. يسجل FP معرّفات دلالية مثل `T03-implementer` و `T03-reviewer`؛ يمكن لوقت التشغيل عرض أي لقب دون تغيير الهوية أو الملكية أو الحداثة أو الدليل.

### هل توفر DeepSeek أو Kimi أو Qwen API نفسها وكلاء فرعيين؟

لا. هذه مزودات نماذج. Kimi Code و Qwen Code هما مضيفا وكلاء بأدوات وكلاء فرعيين خاصة بهما؛ واجهة برمجة التطبيقات للنموذج المستخدمة خلف Claude Code لا تزال تستخدم وقت تشغيل Claude Code. سجل المصدر الرسمي يميز المضيفين عن واجهات برمجة التطبيقات للنماذج فقط ويفشل بالإغلاق عندما تكون القدرة المثبَّتة مفقودة.

### هل يتعلم من جولة واحدة ناجحة؟

يسجل ملاحظة واحدة. حكاية واحدة ليست مخططًا.

### هل هذا ذكاء اصطناعي ذاتي التعديل؟

لا. يمكن للوكلاء في الخلفية اقتراح مرشحين مجمدين. يختبرهم مقيمون مستقلون. الترقية تتطلب صلاحية معلنة، وأدلة آلية، ونافذة ظل مستقبلي، وتراجع.

### هل يحتاج إلى Hermes أو Context7 أو خدمة أخرى قيد التشغيل؟

لا. تم تكييف أفكار بروتوكولاتهم المفيدة في عقد محلي محمول. خوادمهم الخلفية وقواعد البيانات وخدمات MCP والخلفيات الخاصة والمتصفحات ليست اعتماديات.

### هل سيثبِّت FP تلقائيًا MCP مفقودًا؟

فقط بعد موافقة صريحة. يستدعي FP تلقائيًا MCP المطلوب للمهمة والمتاح بالفعل، لكن MCP المفقود يحصل على موجز اكتساب محدود أولاً. الموافقة على التثبيت لا تفوض بصمت تسجيل الدخول، أو الأسرار، أو تغيير الإعدادات، أو إعادة التشغيل، أو الخدمات المقيمة، أو إجراءات المهمة الأوسع.

### هل يعد بعدد أقل من الرموز أو تسليم أسرع؟

فقط بعد أن تثبت مقارنة متكررة مضبوطة ذلك.
